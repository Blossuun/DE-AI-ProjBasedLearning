# E-commerce AI Recommendation Data Pipeline (Micro-batch MVP)

이커머스 환경에서 발생하는 실시간 유저 행동 로그를 수집, 정제하여 AI 추천 모델 및 VIP 예측 모델 학습에 필요한 피처(Feature)를 제공하는 End-to-End 데이터 파이프라인 실습 프로젝트입니다. 

## 🎯 Project Overview
- **Business Context:** 이커머스 플랫폼의 유저 행동(Click, View, Purchase 등) 로그는 실시간으로 대량 발생하며, 오입력이나 시스템 오류로 인한 결측치/이상치(Dirty Data)를 포함하고 있습니다.
- **ML Goal:** AI 모델링 팀은 "특정 유저가 주말에 구매할 확률" 또는 "VIP 고객 예측" 모델을 학습시키기 위해, 노이즈가 제거된 깨끗한 데이터(Clean Data)와 집계된 파생 변수(Feature)를 필요로 합니다.
- **Project Scope:** 본 프로젝트는 실제 프로덕션 환경의 데이터 발생~적재 흐름을 모사하여, 데이터를 정제하고 서빙하는 파이프라인의 MVP(Minimum Viable Product)를 로컬 환경에 구축하는 것을 목표로 합니다.

## 🛠 Tech Stack
- **Language:** Python 3.13
- **Package Management:** uv (초고속 파이썬 패키지 및 프로젝트 관리 도구)
- **Data Processing:** Pandas, NumPy
- **Database:** SQLite (Zero-config RDB)
- **Automation:** Python `schedule`
- **Containerization:** Docker

## 🚀 Pipeline Evolution (과제 진행 기록)

### Task 1: 데이터 생성 및 전처리 (Extract & Transform)

#### 1. 과제 목표
- 실시간으로 들어오는 이커머스 유저 행동 로그를 모사하여, 의도적인 노이즈(결측치, 이상치)가 포함된 원본 데이터를 생성한다.
- AI 모델이 학습할 수 있는 깨끗한 상태로 데이터를 정제하고, 유의미한 파생 변수(Feature)를 추가한다.

#### 2. 도구 선택 및 의사결정
- **데이터 처리: `Pandas`**
  - 현재 실습 환경은 마이크로 배치 당 약 1,000건 내외의 데이터를 처리하므로, 복잡한 분산 처리 프레임워크보다 단일 머신 메모리에서 빠르게 동작하는 Pandas가 가장 적합했습니다.
  - Python 기반의 AI/ML 생태계와 곧바로 호환되며 직관적인 전처리 API를 제공하여 비즈니스 로직 구현에 집중할 수 있었습니다.

#### 3. 마주한 문제 및 해결 방법
- **문제 1: 비정상 데이터로 인한 AI 모델 학습 방해**
  - 행동(Action)이 누락된 로그나 가격(Price)이 음수로 기록된 로그 등 시스템 오류성 데이터가 그대로 유입될 위험이 있었습니다.
  - **해결:** `dropna`로 결측치를 제거하고, 정상적인 가격 범위를 벗어난 데이터(`price < 0` 또는 `price > 450`)를 삭제하는 하드코딩된 방어 로직을 전처리 파이프라인 최전선에 배치했습니다.
- **문제 2: 모델이 이해하기 어려운 날짜 포맷**
  - 원본 타임스탬프(`2026-02-16 14:30:00`) 문자열 자체로는 모델이 주말/평일의 패턴을 학습하기 어렵습니다.
  - **해결:** 날짜 데이터를 파싱하여 주말(토, 일) 여부를 1과 0으로 나타내는 `is_weekend` 컬럼을 생성함으로써, AI 모델링 팀이 즉시 활용할 수 있는 Feature를 제공했습니다.

### Task 2: 데이터 적재 및 통계 추출 (Load & SQL Analysis)

#### 1. 과제 목표
- 메모리(Pandas DataFrame)에서 정제된 데이터를 영구적인 데이터베이스에 적재한다.
- 적재된 원본 로그를 SQL로 조회하여, 모델 학습에 활용할 수 있는 유저별 집계 데이터(Feature)를 추출한다.

#### 2. 도구 선택 및 의사결정
- **데이터베이스: `SQLite`**
  - 로컬 환경에서의 단일 파이프라인 실습이 목적이므로, 별도의 서버 설치나 데몬 설정이 필요한 RDB(MySQL, PostgreSQL 등) 대신 파이썬 내장 라이브러리로 즉시 사용 가능한 SQLite를 선택했습니다.
  - 향후 운영용 DB로 전환하더라도 쿼리 수정이 최소화되도록 표준 SQL 문법을 준수하여 작성했습니다.
- **적재 방식: Pandas `to_sql`**
  - `INSERT` 쿼리를 반복문으로 실행하는 대신, Pandas의 `to_sql` 메서드를 활용해 데이터프레임을 한 번에 적재하여 코드를 간결하게 유지하고 I/O 횟수를 줄였습니다.

#### 3. 마주한 문제 및 해결 방법
- **문제 1: 실행 시마다 데이터가 초기화되는 문제**
  - 초기 테스트를 위해 작성했던 `DROP TABLE IF EXISTS` 구문으로 인해, 파이프라인이 실행될 때마다 이전 데이터가 삭제되었습니다.
  - **해결:** 데이터가 지속적으로 누적되어야 하므로 `CREATE TABLE IF NOT EXISTS`로 DDL을 수정하고, `to_sql` 옵션을 `if_exists='append'`로 변경하여 기존 테이블에 데이터가 추가되도록 파이프라인 로직을 개선했습니다.
- **문제 2: 단순 적재를 넘어선 정보 추출 필요**
  - AI 모델은 개별 로그가 아닌 특정 기준(유저, 시간대 등)으로 요약된 데이터를 필요로 합니다.
  - **해결:** SQL의 `GROUP BY`와 집계 함수(`SUM`, `COUNT`)를 활용하여 유저별 총 구매액(VIP 스코어)을 구하고, `WHERE` 절을 통해 특정 조건(주말 발생 구매)의 데이터를 필터링하는 쿼리를 작성하여 분석 모듈(`analyzer.py`)을 구성했습니다.

### Task 3: 파이프라인 자동화 (Automation)

#### 1. 과제 목표
- 단발성으로 실행하던 파이썬 스크립트를 주기적으로 자동 실행되도록 구성하여, 지속적으로 데이터가 유입되는 환경을 모사한다.

#### 2. 도구 선택 및 의사결정
- **스케줄러: Python `schedule` 라이브러리**
  - OS에 종속적인 `cron`을 사용하면 윈도우/맥/리눅스 등 환경이 바뀔 때마다 스케줄링 설정도 새로 해야 하는 번거로움이 있습니다.
  - 파이썬 코드 내에서 동작하는 `schedule` 라이브러리를 도입하여, 실행 환경(OS)에 구애받지 않고 단일 진입점(`main.py`)만 실행하면 스케줄러가 알아서 동작하도록 구성했습니다.

#### 3. 마주한 문제 및 해결 방법
- **문제 1: 지속적인 데이터 파이프라인 시뮬레이션**
  - 실시간으로 끊임없이 들어오는 이커머스 트래픽을 로컬에서 어떻게 흉내 낼 것인지 결정해야 했습니다.
  - **해결:** 스케줄링 주기를 10초 단위로 짧게 설정하여, 잘게 쪼개진 데이터를 주기적으로 처리하는 마이크로 배치(Micro-batch) 형태로 동작하게 만들었습니다.
- **문제 2: 프로세스 중단 위험 (예외 처리 부재)**
  - 스케줄러가 무한 루프(`while True`)로 동작하는 도중, 1회의 실행에서 예상치 못한 에러가 발생하면 파이썬 프로세스 전체가 종료되어 스케줄러가 멈추는 문제가 있었습니다.
  - **해결:** 실행 로직 전체를 `try-except` 블록으로 감싸, 특정 주기에 에러가 발생하더라도 로그만 출력하고 프로그램은 종료되지 않은 채 다음 10초 뒤의 실행을 대기하도록 안정성을 높였습니다.

### Task 4: 실행 환경 격리 및 컨테이너화 (Containerization)

#### 1. 과제 목표
- 파이프라인 스크립트가 로컬 PC의 OS나 파이썬 설치 환경에 종속되지 않고, 어느 서버에서든 동일하게 실행될 수 있도록 패키징한다.

#### 2. 도구 선택 및 의사결정
- **환경 격리: `Docker` 및 `python:3.13-slim`**
  - 파이썬 가상환경(`venv`)만으로는 해결할 수 없는 시스템 레벨의 라이브러리 차이를 극복하기 위해 Docker를 도입했습니다. 불필요한 패키지가 빠져 용량이 작은 `slim` 이미지를 베이스로 선택했습니다.
- **패키지 관리: `uv`**
  - 기존 프로젝트에서 사용 중이던 `pyproject.toml`과 `uv.lock`을 그대로 활용하여, `pip` 대비 압도적으로 빠르고 결정론적(Deterministic)인 환경 구축이 가능한 `uv`를 Dockerfile 내에 통합했습니다.

#### 3. 마주한 문제 및 해결 방법
- **문제 1: 백그라운드 실행 시 터미널 로그 출력 불가 현상**
  - `docker run -d` 옵션으로 컨테이너를 백그라운드 실행했을 때, 파이프라인은 정상 동작함에도 불구하고 `docker logs` 화면에 파이썬의 `print` 로그가 찍히지 않았습니다.
  - **해결:** 파이썬의 표준 출력 버퍼링(Output Buffering) 기능이 원인임을 파악하고, Dockerfile에 `ENV PYTHONUNBUFFERED=1` 옵션을 추가하여 메모리에 텍스트를 모아두지 않고 즉시 출력하도록 강제했습니다.

### Task 5: 데이터 영속성 확보 (Data Persistence)

#### 1. 과제 목표
- 도커 컨테이너가 종료되거나 삭제되더라도, 그동안 수집하고 정제하여 적재한 데이터베이스 파일(`pipeline.db`)이 유실되지 않도록 보존한다.

#### 2. 도구 선택 및 의사결정
- **영구 저장 방식: Docker Bind Mount (`-v`)**
  - Docker Named Volume을 사용할 수도 있지만, 로컬 환경에서 개발과 검증을 직관적으로 수행하기 위해 호스트 PC의 특정 디렉토리와 컨테이너 내부의 디렉토리를 직접 연결하는 Bind Mount 방식을 선택했습니다. 이를 통해 로컬에 설치된 DBeaver 같은 툴로 컨테이너 내부의 DB 파일을 즉시 열어볼 수 있습니다.

#### 3. 마주한 문제 및 해결 방법
- **문제 1: 컨테이너의 휘발성(Ephemeral)으로 인한 데이터 유실**
  - 도커의 격리된 파일 시스템 특성상, 파이프라인 업데이트나 재시작을 위해 컨테이너를 삭제(`docker rm`)하면 그 안의 데이터베이스도 함께 날아가는 치명적인 문제가 발생했습니다.
  - **해결:** 컨테이너 실행 명령어에 `-v` 옵션을 추가하여, 호스트 머신의 `data/` 폴더를 컨테이너의 `/app/data/` 경로에 마운트했습니다. 컨테이너를 강제로 지웠다 다시 띄워도 기존 데이터가 온전히 남아 누적되는 것을 검증했습니다.

### Task 6: 클라우드 데이터 레이크 연동 (AWS S3 Integration)

#### 1. 과제 목표
- AI 모델링 팀이 대규모 학습을 수행할 수 있도록, 정제된 원본 로그 데이터를 클라우드 오브젝트 스토리지(Data Lake)에 적재한다.

#### 2. 도구 선택 및 의사결정
- **저장소: `AWS S3` 투트랙(Two-track) 아키텍처**
  - 분석용 집계 데이터는 기존 SQLite(RDB)에 저장하고, AI 학습용 원본 로그는 AWS S3에 저장하도록 역할을 분리했습니다. 대규모 I/O 작업 시 RDB에 발생하는 부하를 방지하고, 스토리지 비용을 최적화하기 위한 결정입니다.
- **포맷: `Parquet` vs CSV**
  - CSV 대비 압축률이 높고 읽기 속도가 빠른 컬럼 기반 스토리지 포맷인 `.parquet`을 채택하여 S3 스토리지 비용 및 네트워크 전송 비용을 절감했습니다.

#### 3. 마주한 문제 및 해결 방법
- **문제 1: 클라우드 자격 증명(Credential) 보안 취약성**
  - AWS Access Key를 소스 코드에 하드코딩할 경우, Git 버저닝 과정에서 키가 유출되는 심각한 보안 사고로 이어질 수 있습니다.
  - **해결:** `python-dotenv` 패키지를 도입하고 `.env` 파일을 통해 환경 변수로 키를 분리했습니다. 이후 도커 컨테이너 실행 시 `--env-file` 옵션을 사용하여, 코드 레벨의 노출 없이 안전하게 컨테이너 내부로 자격 증명을 주입하는 방식을 구현했습니다.

### Task 7: AI 예측 모델 학습 파이프라인 (Downstream ML Pipeline)

#### 1. 과제 목표
- 데이터 엔지니어링 파이프라인이 S3 데이터 레이크에 적재한 원본 `.parquet` 데이터를 활용하여, 유저의 과거 행동 기반 VIP 여부를 예측하는 머신러닝 모델을 학습하고 저장한다.

#### 2. 도구 선택 및 의사결정
- **데이터 로드: Pandas & `s3fs`**
  - S3에 저장된 다수의 Parquet 파일을 로컬 메모리로 효율적으로 병합하여 가져오기 위해, Pandas의 `read_parquet`과 S3 파일 시스템 인터페이스인 `s3fs`를 결합하여 사용했습니다.
- **머신러닝 알고리즘: `scikit-learn` (Random Forest)**
  - 행동 로그의 집계 데이터(Tabular data)를 분류하는 데 있어 딥러닝(Deep Learning)을 바로 도입하는 것은 모델의 설명 가능성(Explainability) 저하와 불필요한 연산 리소스 낭비를 초래합니다. 베이스라인 모델로서 앙상블 트리 기반의 Random Forest를 선택하여 과적합을 방지하고 피처 중요도를 직관적으로 파악할 수 있도록 설계했습니다.

#### 3. 마주한 문제 및 해결 방법
- **문제 1: S3 데이터 레이크의 분할된 파일(Partitioning) 구조 읽기**
  - 마이크로 배치로 인해 S3에 10초 단위로 수많은 파일이 조각나서 저장되므로, 이를 한 번에 학습 데이터셋으로 구성해야 했습니다.
  - **해결:** `boto3.client.list_objects_v2`를 통해 버킷 내 객체 리스트를 가져온 뒤, 순회하며 데이터프레임 리스트에 담아 `pd.concat`으로 메모리 병합하는 구조를 채택했습니다. (데이터 볼륨이 단일 메모리 한계를 초과하게 될 경우 분산 프레임워크로 교체할 여지를 둠)
- **문제 2: 모델 학습 목적에 맞는 피처 엔지니어링 (Data Leakage 방지)**
  - VIP 예측 모델의 타겟(y)을 '총 구매액(total_spend)'으로 정의했는데, 학습 피처(X)에 이 구매액이 그대로 들어가면 정답을 미리 알려주는 데이터 누수(Data Leakage)가 발생합니다.
  - **해결:** 모델 학습 데이터(X) 구성 시 `total_spend` 컬럼을 명시적으로 제외하고, 구매 이전 단계의 깔때기(Funnel) 지표인 `click_count`, `view_count`, `cart_count` 등 행동 데이터만을 활용하도록 코드를 설계했습니다.
